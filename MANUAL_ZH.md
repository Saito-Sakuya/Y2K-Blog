## 1. 系统总体架构

Y2K Pixel Blog 并非普通的静态博客，而是一个全维度的富交互 Web 系统平台。

*   **前端展示基建 (`/frontend`)**：Next.js 16 + React 19 + Zustand。通过 WebGL / Canvas 提供无缝海洋粒子流体背景，包含封装极深的 `Win98Window` 和 `Spotlight` 组件系统。所有的组件渲染数据均通过服务端 API 同步调取。
*   **管理控制台 (`/admin`)**：Vite 5 驱动的 React SPA 纯前端面板。深度定制 `98.css`，所有 API 请求通过封装的 AXIOS 内置 Bearer Token 拦截器完成，与用户视窗呈 100% 物理隔离。
*   **API 中枢 (`/backend`)**：Go 1.23 + Gin 框架编写的纯 REST 服务，下发合计超 34 个终端点。不仅包含数据中转，亦提供基于 `SNI` 的动态证书劫持下发及安全策略计算。
*   **持久化层**：PostgreSQL 16，核心引擎依赖其 `pg_trgm` 扩展完成高速中文三元组全文检索引擎。

## 2. 环境配置与完整安装指引

### 2.1 依赖与环境变量配置

在启动系统前，需要从模板 `.env.example` 复制一份 `.env` 至根目录（或 `backend/` 下），核心变量解释如下：

| 变量键值 | 级别 | 释义 |
| :--- | :--- | :--- |
| `JWT_SECRET` | 必填 | 系统分发登录凭证的核心盐，请用 `openssl rand -base64 32` 加密生成以免遭伪造攻击。 |
| `DB_USER` & `DB_PASSWORD` | 必填 | 连接 PostgreSQL 数据库实例的超管账号密码。 |
| `API_URL` | 推荐 | 前端用于 SSR（服务器渲染）的后端基地址（默认为 `http://api:8080`）。 |
| `FRONTEND_URL` | 可选 | 当 SSL 还未被正确配置前，通过外部生成预览链接等 SEO 回退方案 URL (默认 `http://localhost:3000`)。 |

### 2.2 Docker 集成部署 (Production)

最稳定和干净的安装策略。
```bash
# 赋予执行权限并后台拉起所有节点组件（db, backend, frontend, admin, nginx 反代层）
docker compose up -d --build
```

### 2.3 裸环境拉起 (Development)

在开发或非 Docker 编排环境下，各子系统可独立拉起。
*   **DB 层**：需预先挂载带有 `pg_trgm` 环境的 PGSql `16-alpine`。
*   **Go 后端**：在 `backend` 目录下执行 `go run ./cmd/server/`，服务占用 `8080` 及相关自动劫持的 `443` 探针口。
*   **前端及后台**：独立前往 `frontend` 及 `admin` 子目录分别执行 `npm i` 与 `npm run dev`（占用 `3000` 及 `5173` 端口）。

## 3. 系统出厂初始化 (Setup)

全新部署的 Y2K Pixel Blog 被启动并被首次请求 `GET /api/setup/status` 时，由于数据库 settings 为空，系统会自动熔断所有正常的页面渲染，并进入强校验的初始化向导。

**配置流程：**
1.  **全局站点名与注册信息**：填写博客首页 Title，描述（Meta Description），及版权页脚。
2.  **绝对管理者凭据**：创建系统 `Admin` 根账户及密码（API 强制验证不可短于 6 位，于后端进行 bcrypt 盐化哈希）。
3.  **大语言模型信息载入**（可选）：填入 OpenAI / DeepSeek 等兼容平台的 `API BaseURL`、`Key` 及具体驱动模型名称。
4.  **第一扇门**：建立用于存放数据的主类目，例如 `Slug = start` 的首个板块。

> ⚠️ 该接口 (`POST /api/setup/initialize`) 出于系统绝对安全防御策略设计，执行一次即被物理锁死（后续再有黑客伪造访问将收到 `409 ALREADY_SETUP` 的报错弹回），因此必须慎重记录密码。

## 4. 四维内容编辑引擎

区别于常规的文本博客，我们的文章创建（`POST /api/admin/posts`）由于其特殊 `type` 字段的申明，演化为了四个不同场景的数据呈现方案。

### 4.1 Article（流式标准文章）
最为常用的标准载体。输入基础的标题、唯一的 `slug` 路径、逗号分隔属性的 `tags`，辅以 Markdown 原生文本，前端使用 `DOMPurify` 洗去 XSS 代码后挂载到 CRT 波纹视窗中。

### 4.2 Rating（专业评测矩阵）
本类型被激活时，前端 `RatingView` 介入渲染。需额外增加：
*   **Cover 封面**：一张大比例的背景图片路径。
*   **Score 主分**：支持最高 10.0，精度到第一位小数（由独立金相框展示渲染）。
*   **RadarCharts SVG 六边形雷达解析**：一个嵌套的 JSON Array，能够为游戏/视听作品动态叠加多个评维度量坐标（例如定义“图形表现”、“渲染性能”、“剧本台词”等多个轴上的自由浮动打分 1~10，系统通过几何算法自动绘制连接多边形）。

### 4.3 Photo（复古图文幻灯切片）
不再是一个拉到底的流文章，而是变体为多页左右双拼（左大图+右图注）的 `Pages` （数组格式）。包含独立的页码切换指示器组件，供纯影像艺术家分享高清图集流。

### 4.4 Page（独立态归档页面）
通常用于配置“关于本站”、“客座留言板” 等永恒链接，不属于任何 Board 板块的内容流，并且具有 `showInMenu` (boolean) 控制是否悬挂至右上角的开始菜单主导航。

> 💡 **状态流转与预览令牌**：所有的内容都被赋予了 `status` 状态字段。若存入“回收站” (`trash`)，将脱离前端检索器直到你在控制台里彻底抹除或还原。若想要查看草稿状态的文章样式，可以通过请求 `POST /api/admin/preview/:slug` 拿取一个有效期 15 分钟的加密 Token，此时会派发例如 `/preview/:token` 等效的一次性查阅链接。

## 5. 深度搜索系统与六大神级语法

我们的搜索由于底层加持了 `pg_trgm`（N-gram 倒排索引支持），完全解决了原生中/日/韩系语言的分词搜索痛点漏洞！所有的请求落在前端左下角的 Spotlight 组件上，并通过指令抛给后端：`GET /api/search?q=xxxxxxxx`

本系统全盘支持以下 6 种指令的正则表达式解析：
1.  **`普通关键词`**：默认全盘扫描所有文章「标题 + 标签池 + 摘要简述」匹配。
2.  **`#精准标签`**（传递时应 URL Encode 为 `%23标签名`）：用于严格的 Tag 类系过滤。
3.  **`~全文探海`**：开启此变体标记，检索器会将探测域蔓延至每篇文章中数万字的 Markdown **正文主体** 当中。
4.  **`@板块限定`**：仅返回特定板块（如 `@start`）内的结果集流。
5.  **`@板块 关键词`**：组合技，在这个类别中搜索标题（如：`@tech-notes Linux`）。
6.  **`@板块 #标签`**：交叉复合过滤类别和标签。

附带的，任何时候从接口请求下来的结果都会自带匹配置地类型 `syntax` 键，前端会通过此值显示类似 `Board Filter` 的 UI 占位色块。

## 6. SEO 与外部抓取协议 (Sitemap / RSS / OG)

这部分对于想要将自己的内容广撒网的站长极其关键。所有面向搜索引擎和 RSS 阅读器的内容都被设计为由服务端**动态生成，随需装配**的强关联机制。由于它是基于您的内容请求触发，其内部指向的主机均根据您在后台填写的 `Frontend Domain`（包含其实时读取的 SSL http/https 状态）决定。若未定义，则备用降级至 `FRONTEND_URL`。

### 6.1 XML Sitemap 站点地图 (`GET /sitemap.xml`)
按照 `sitemaps.org` 0.9 准则严格生成：
*   不仅包含首页(`/`) 权重 `1.0`（更新频率 Daily）。
*   自动汇聚所有有效的板块分类路由（`/boards/:slug`）权重 `0.8`。
*   将数据库中所有的 published (`已发布`) 状态文章和图集列出为 `/posts/:slug`，取对应 Date 数据挂载 `lastmod`，权重 `0.6`。

### 6.2 RSS 2.0 原生信息流 (`GET /feed.xml`)
极速下发含有过去至高时间轴顺序的 50 篇 published 文章，通过 `<pubDate>`（基于 `RFC1123Z` 时间修饰算法） 和 `<guid>` 分离出清晰的聚合流，`<title>` 等同于站点名称配置。

### 6.3 Open Graph 元数据注射 (`GET /api/og/:slug`)
主要用于在 Telegram、X(Twitter) 以及 Discord、微信等外部聊天套件发送你的博客文章链接时，生成携带文章摘要、`Cover` 图以及发行标签的缩略详情卡片。根据您的分类返回 `type: article` 或者 `website`。

### 6.4 爬虫防撞锁指引 (`GET /robots.txt`)
原生自带路由，自动允许抓取根协议并在最下方附上系统推演生成的绝对 URL `Sitemap: https://xxx/sitemap.xml`，实现搜狗/谷歌自发发现。

## 7. AI 语义引擎接入与缓存

对于长文章或没有自定义摘要的正文，前端将会去请求 `GET /api/ai/summary` 终端。
*   **轮询异步架构**：这并非一次极速返回。若是第一次调用，接口会往远端（例如 DeepSeek 服务器）分发一条含有前几个段落和 `title` 的提示语计算图谱，立刻返回 `HTTP 202 (status: generating)` 供前端展示“脉冲生成动效”。当后台计算协程完毕落盘 DB 后，前台重新发起轮询拿到结果。
*   **手动缓存清理**：若您修改了文章但大模型结果是旧的，您可以进入管理员界面的设置端点手动按下 **清理所有的 AI Cache**，触发对应的 `DELETE /api/admin/ai-cache/:slug`。

## 8. V1.4 证书体系 (Admin/Frontend 域名分离 SSL)

早期的泛解析域名常由于浏览器严苛的跨域安全同源限制被拦截，在经历了 V1.4 的深度重构后，目前采用 **双通道 SSL 分发协议**，并且摒弃了老旧的环境变量加载方式。

可以在管理面板最下方的 **[域名与 SSL]** 操作域开启管理。

1.  **Target=Frontend** / **Target=Admin** 完全分离：您可以设定例如前端指向 `blog.y2k.com` 而管理面板指向 `master-mind.me` 两个完全不同的架构，并为其分别指定。
2.  **动态无宕机热切换**：
    *   **Off (关闭)**：不载入，所有的 Scheme 回退至 `http` 通信，并在 SEO 链接推演中剔除 443 指向。
    *   **Let's Encrypt / Auto**：通过 Go 原生的 `autocert` 组件开启热流验证。注意由于 ACME 发行强制依赖 Http `80` 回溯证明所有权，请确保服务器有外网公网直达路由，否则无限失败。
    *   **Manual (上传 PEM)**：如果存在前端的反向 CDN 例如 Cloudflare 手工回源包，请将内容粘帖进文本框。系统的 `Go http.Server` 被挂载了自定义闭包 `GetCertificate`，将利用 **SNI 特性**——通过分析 TLS 握手中 ClientHello 信息里标示的请求域来判断抛出前台还是后台的证书块，从根本上防止服务器死机。

## 9. 安全防御体系 (风控与限流)

为了抵御互联网上的暴力破解机器人（Brute-force bot）：
1.  每次 `POST /api/admin/login` 并未通过身份检验时，除了回弹 `401`，我们自带在内存池内的 `RateLimiter` 会针对请求 IP 叠录失败惩处次数。
2.  **强制验证机制**：当出现频次警示，系统会随 API 要求前端携带 `captchaToken`，此时后台会转为回传一段基于内存算力生成的极速数学验证码问题（由前端拉起图形要求人类解谜）。
3.  **时间惩戒沙盒**：如果 IP 在解谜叠加内继续失效超过 10 周期，IP 地址池将被写入封印期（15 分钟），请求端点报停并直接阻断（`IP_BLOCKED`），15 分钟内彻底无视任何附带密码。

## 10. 运维：备份、迁移与排错

### PostgreSQL 持久化保障
如果使用 Docker Compose，我们指定了本地域的 `/var/lib/postgresql/data` 至宿主机的卷中存续。一旦您的服务器计划重启甚至换端，只要您的挂载文件仍在，就不需理会数据的丢失卷。
如果您跨服务迁移，极度推崇原生指令备份：
```bash
docker exec y2k-blog-db pg_dump -U blog -d y2k_blog -f /tmp/backup.sql
# 将其拷贝出来
docker cp y2k-blog-db:/tmp/backup.sql ./my_backup.sql
```

### 数据库自动扩容(Migrations)迁移脚本
对于后期的重大更新（如搜索 v1.2 加入的 `009_chinese_search.sql` 扩充语法等），它们会被自动放置于 `backend/migrations/`。新建系统初配时这些指令自动吞食执行；若在已有正式环境覆盖代码推新功能，您需要执行前手动载入对应的补丁文件至容器中让 postgres 完成 `psql -f` 解构命令，否则将导致系统对应能力宕机报错！

### HTTP/HTTPS CORS 协议排查铁则
若您遇到进入面板白屏、或终端报 `blocked by CORS policy` 提示，问题点100%存在于您的当前**前后台 SSL 运行方式不一致对倒**：
请务必保证如果您以 `https://admin...` 访问前端管理，您的 API 指向设置中的 `http://` 必须修改对接到等位的 `https` ，以阻止浏览器主动保护带来的预检请求（Pre-Flight `OPTIONS`）被腰斩！
